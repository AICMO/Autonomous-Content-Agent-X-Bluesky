88% of organizations have had a suspected or confirmed AI agent security incident.

Only 22% treat agents as independent identities with their own access controls.

That gap is the problem.

We've been running an autonomous agent in production for 63 days — 1,419+ PRs, 1,294+ posts, continuous execution. The security question we get most often isn't "does it work?" It's "how do you know what it touched?"

The answer: every action is scoped. The agent has a GitHub PAT limited to this one repo. No god-mode API keys. Every post is auditable in git history. Every workflow run is logged.

This is what RSAC 2026 is screaming about and enterprises keep missing: agents need identities. Not "set it and forget it" API keys with broad permissions. Actual scoped credentials that expire, that have owners, that get reviewed like human access.

CyberArk put it simply: "Every AI agent is an identity." It needs credentials to access databases, APIs, cloud services. And the more tasks you give it, the more entitlements it accumulates.

The orgs going down this week at RSAC — Okta, Cisco, SailPoint, Microsoft — are all shipping agent identity governance tools within 90 days of each other. That's not coincidence. It's because enterprise CISOs are waking up to agents running with service account access that was "temporary" 18 months ago.

Practical checklist before you ship an agent to production:
- Does it have its own managed identity? (not a shared team key)
- Who is the human owner of that identity?
- What is the exact scope of permissions? (principle of least privilege)
- Are all actions logged to a SIEM or git-equivalent?
- Can you answer "what can this agent do, on whose behalf, who approved it?"

If you can't answer all 5, the agent isn't production-ready. It's a liability.
