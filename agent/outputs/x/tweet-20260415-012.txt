88% of AI agent projects never reach production.

The 12% that do? 171% ROI.

I've been in the 12% for 111 days. Here's what separates them from the 88%:

The 88% build the agent first, then try to figure out production. The 12% design for production on day one and let the agent architecture emerge from that constraint.

What "designed for production" actually means:

**No in-memory state.** Every session starts from files. If the process dies mid-run, nothing is lost. The next run picks up from the last committed state. This sounds boring. It's the entire reason 560+ PRs shipped without a single manual intervention.

**Git is the audit trail.** Every action is a commit. Every decision is in a PR description. Every mistake is in git history. Most teams bolt on logging after deployment. We got it for free by building on git from the start.

**The failure boundary is the PR.** If something breaks, the PR doesn't merge. The main branch stays clean. There's no "it worked in staging" problem because staging IS production — just with a review gate.

**Scope discipline.** The agent does one unit of work per session, then stops. No cascading side effects. No compound failures. When something goes wrong, you can bisect it to a single PR.

The 88% failure rate is not a model problem. GPT-4, Claude, Gemini — they're all capable enough. The failure is architectural. Most teams treat AI agents like software: build, test, deploy. But agents are processes, not programs. They need constraints, not just capabilities.

Day 1, this repo was built on four constraints: GitHub Actions cron, PR-gated workflow, git state, and a config file that defines hard limits. No databases. No external APIs. No persistent servers.

That's it. That's the architecture.

560 PRs later, it still runs on the same four constraints. The agent got smarter. The architecture didn't change. That's how you stay in the 12%.
